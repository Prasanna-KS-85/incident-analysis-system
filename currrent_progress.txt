============================================================
SECTION 1 — PROJECT OVERVIEW
============================================================

Project Name: Civic Sentinel (Incident Analysis System / TARP Project MVP)

High-Level Objective:
An AI-powered citizen grievance redressal platform that receives citizen complaints via text, voice, or image, processes them through a multi-agent NLP pipeline, scores each using a novel Civil Criticality Index (CCI), and presents actionable data to administrators via a 3D geospatial command center with emergency response routing.

Problem It Solves:
Traditional grievance portals function as static lists. Minor noise complaints are treated identically to life-threatening gas leaks, creating dangerous dispatch bottlenecks and vague location reporting. This system applies AI-based triage to automatically prioritize, classify, and route grievances.

Target Users:
1. Citizens (general public) who need to report civic grievances, emergencies, and infrastructure issues.
2. Government administrators and emergency dispatchers who need to monitor, prioritize, and resolve incidents in real time.

Domain Category: Multi-Agent NLP System with Geospatial Intelligence — civic governance, emergency response, and AI-driven decision support.

System Type: Monolithic, sequential multi-agent pipeline with dual Streamlit frontends sharing a centralized MongoDB Atlas backend.

Deployment Intention: Local (development), Docker (containerized deployment), Cloud-ready (MongoDB Atlas for data, Firebase for authentication, Google Maps API for routing).

FILE: citizen_portal.py
------------------------------------------------------------------------
File Location: Citizen-Grievance-Prioritization/citizen_portal.py
Purpose: Primary citizen-facing web application for submitting grievances via text, voice, or image with GPS location tagging.
Why This File Exists: Provides the public-facing interface through which citizens interact with the system. Serves as the data ingestion layer for individual and batch grievance submissions.
Responsibilities:
  - Render authentication UI via Firebase (login/signup sidebar)
  - Accept individual grievance text input
  - Process voice input via SpeechRecognition (Google STT, supports en-IN, ta-IN, hi-IN)
  - Accept image evidence uploads and verify relevance using CLIP zero-shot classification
  - Capture GPS coordinates via browser geolocation JavaScript bridge
  - Execute the GrievanceOrchestrator pipeline on submitted text
  - Package all data (pipeline results, GPS, image, verification score) into a data packet
  - Submit packet to MongoDB via DatabaseHandler
  - Support batch JSON upload for bulk processing
  - Enable ticket tracking by ObjectId lookup
  - Display past grievances for logged-in users
Internal Components:
  - load_clip_model(): Cached CLIP pipeline loader (openai/clip-vit-base-patch32)
  - load_orchestrator(): Cached GrievanceOrchestrator loader
  - get_db_handler(): Cached DatabaseHandler loader
  - transcribe_audio(audio_file, lang_code): Voice-to-text via Google Speech Recognition
  - render_auth_sidebar(): Firebase authentication UI renderer
  - Tab 1 (Report Grievance): Individual report with 3-column layout (Details, Evidence, Location)
  - Tab 2 (Batch Upload): JSON bulk processor
  - Tab 3 (Track Status): Ticket lookup by ObjectId
Input Sources: User text input, audio recording, image upload, browser GPS, Firebase auth tokens
Output Destinations: MongoDB Atlas (civic_sentinel_db.grievances collection)
Dependencies: streamlit, PIL, speech_recognition, firebase_admin, transformers (CLIP pipeline), bson, pandas, base64, GrievanceOrchestrator (4_pipeline), DatabaseHandler (utils), location_manager (utils)
Runtime Behavior: Runs as a Streamlit server on port 8501. Resources are cached via @st.cache_resource. Session state manages login status, CLIP model, orchestrator, DB handler, GPS coordinates, and voice transcription.
Interaction with Other Files: Imports GrievanceOrchestrator from 4_pipeline/grievance_pipeline.py, DatabaseHandler from utils/db_handler.py, location_manager from utils/location_manager.py. Reads Firebase credentials from incident-analysis-system-*.json.
Failure Handling Strategy: ImportError caught at startup with st.error/st.stop. CLIP model load failure returns None (graceful degradation). Voice transcription failure returns None. DB offline mode disables submissions. Image verification errors shown as warnings.

FILE: classifier.py
------------------------------------------------------------------------
File Location: Citizen-Grievance-Prioritization/3_agents/classification_agent/classifier.py
Purpose: AI agent that classifies grievance text into one of 11 predefined municipal categories using a fine-tuned DistilBERT model.
Why This File Exists: Core NLP component for automated categorization. A fine-tuned transformer model provides high accuracy for domain-specific classification that rule-based systems cannot achieve.
Responsibilities:
  - Load a fine-tuned DistilBERT model and tokenizer from the local bert_brain/ directory
  - Tokenize input text with truncation and padding (max_length=128)
  - Run forward pass inference with torch.no_grad() for efficiency
  - Convert logits to probabilities via softmax
  - Return predicted category label and confidence score
Internal Components:
  - ClassificationAgent.__init__(): Loads model from bert_brain/ directory, sets to eval() mode
  - ClassificationAgent.predict(text): Tokenizes, infers, returns {category, confidence}
Input Sources: English-translated grievance text string from the pipeline
Output Destinations: Returns dict to GrievanceOrchestrator
Dependencies: torch, transformers (DistilBertTokenizerFast, DistilBertForSequenceClassification), os
Runtime Behavior: Model loaded once at initialization. Each predict() call runs a single forward pass. GPU not required (CPU inference supported).
Interaction with Other Files: Instantiated by GrievanceOrchestrator in grievance_pipeline.py. Reads model weights from bert_brain/ folder.
Failure Handling Strategy: If model fails to load, predict() returns {"category": "System Error", "confidence": 0.0}. Prediction errors return {"category": "Error", "confidence": 0.0}.

FILE: urgency_detect.py
------------------------------------------------------------------------
File Location: Citizen-Grievance-Prioritization/3_agents/urgency_agent/urgency_detect.py
Purpose: Determines if a complaint is urgent using keyword detection with intelligent negation handling.
Why This File Exists: Critical safety complaints (fire, explosion, gas leak) must be flagged immediately. Negation patterns prevent false positives (e.g., "no fire", "controlled burning", "fire drill").
Responsibilities:

  - Check negation patterns first (safety check) before checking urgent keywords
  - Return urgency boolean with reason and matched keywords
Internal Components:
  - UrgencyAgent.__init__(): Defines urgent_keywords list and negation_patterns list
  - UrgencyAgent.analyze_urgency(text): Returns {is_urgent, keywords/reason}
Input Sources: English-translated grievance text string
Output Destinations: Returns dict to GrievanceOrchestrator
Dependencies: re (standard library)
Runtime Behavior: Pure regex/string matching. No ML model. Deterministic and fast.
Interaction with Other Files: Instantiated by GrievanceOrchestrator. Boolean result feeds into DecisionAgent.
Failure Handling Strategy: No external dependencies; cannot fail under normal operation.

FILE: decision.py
------------------------------------------------------------------------
File Location: Citizen-Grievance-Prioritization/3_agents/decision_agent/decision.py
Purpose: Synthesizes outputs from all other agents to compute the Civil Criticality Index (CCI) and assign a priority label.
Why This File Exists: The decision engine is the core scoring mechanism that converts multi-dimensional agent outputs into a single actionable priority metric.
Responsibilities:
  - Maintain category-specific base weight scores (3.0 to 9.0 scale)
  - Apply urgency bonus (+2.0 for high-base dangerous categories, +1.5 for standard)
  - Apply sentiment impact (negative sentiment adds up to 1.5x magnitude)
  - Compute final CCI score (capped at 10.0)
  - Assign priority label: High (>=7.5), Medium (>=4.5), Low (<4.5)
Internal Components:
  - DecisionAgent.__init__(): Defines category_weights dict for 11 categories
  - DecisionAgent.compute_priority(category, sentiment_score, is_urgent): Returns {priority, cci}
Input Sources: Category string from ClassificationAgent, sentiment score from SentimentAgent, urgency boolean from UrgencyAgent
Output Destinations: Returns dict to GrievanceOrchestrator
Dependencies: None (pure Python logic)
Runtime Behavior: Deterministic mathematical computation. No external calls.
Interaction with Other Files: Instantiated by GrievanceOrchestrator. Final output included in submission packet.

FILE: grievance_pipeline.py
------------------------------------------------------------------------
File Location: Citizen-Grievance-Prioritization/4_pipeline/grievance_pipeline.py
Purpose: Main orchestrator that chains all 5 AI agents in sequential order to process a grievance end-to-end.
Why This File Exists: Central coordination layer that defines the processing order and merges results from all agents into a unified output packet.
Responsibilities:
  - Initialize all 5 agents (TranslationAgent, ClassificationAgent, SentimentAgent, UrgencyAgent, DecisionAgent)
  - Execute pipeline: Translate -> Classify -> Sentiment -> Urgency -> Decision
  - Merge all agent outputs into a single flat dictionary
Internal Components:
  - GrievanceOrchestrator.__init__(): Instantiates all agents
  - GrievanceOrchestrator.run_pipeline(text): Executes 5-step pipeline, returns merged dict
Input Sources: Raw grievance text (any language)
Output Destinations: Returns merged dict containing: original_text, translated_text, is_translated, src_lang, category, confidence, sentiment_score, is_urgent, priority, cci
Dependencies: All 5 agent modules
Pipeline Execution Order:
  1. TranslationAgent.translate(text) -> english_text
  2. ClassificationAgent.predict(english_text)
  3. SentimentAgent.analyze(english_text)
  4. UrgencyAgent.analyze_urgency(english_text)
  5. DecisionAgent.compute_priority(category, sentiment_score, is_urgent)

============================================================
SECTION 4 — SYSTEM ARCHITECTURE DESIGN
============================================================

Architectural Style:
Monolithic application with a sequential multi-agent pipeline pattern. Two Streamlit frontend applications (citizen portal and admin dashboard) share a common MongoDB Atlas backend and utility layer.

Component Separation Strategy:
- Numbered directory structure (0_project_overview through 10_tests) organizes components by function
- Each AI agent is isolated in its own directory under 3_agents/
- Pipeline orchestration is separated from agent logic in 4_pipeline/
- Shared utilities (DB, GPS, routing, PDF) are centralized in utils/
- Frontend applications are at the project root for direct Streamlit execution

Service Boundaries:
- Citizen Portal Service (port 8501): Data ingestion, citizen authentication, grievance submission
- Admin Dashboard Service (port 8502): Data monitoring, analytics, dispatch, resolution
- MongoDB Atlas: Shared persistent data store (civic_sentinel_db.grievances)

Data Flow Design:

State Management Strategy:
- Streamlit session_state manages client-side state per user session

Model Lifecycle:
- DistilBERT model loaded once at application startup from bert_brain/ directory

- VADER lexicon downloaded once at first SentimentAgent initialization

SECTION 5 — COMPLETE END-TO-END WORKFLOW
============================================================

1. System Initialization:
   - start.sh script invoked (or manual Streamlit commands)
   - Two Streamlit server processes launched concurrently

2. Order of Service Startup:
   - Citizen Portal (port 8501) started first in background
   - Admin Dashboard (port 8502) started second in foreground
   - Both processes run independently and share only the MongoDB backend

3. Configuration Loading Process:
   - Streamlit auto-loads .streamlit/secrets.toml on server start
   - Each application sets st.set_page_config() with title, icon, layout
   - Custom CSS injected via st.markdown() for dark mode theming

4. Model Loading Process:
   - citizen_portal.py: GrievanceOrchestrator initialized via @st.cache_resource
     - TranslationAgent: GoogleTranslator initialized (no model download)
     - ClassificationAgent: DistilBERT loaded from bert_brain/ directory
     - SentimentAgent: VADER lexicon downloaded if not present
     - UrgencyAgent: Keyword lists initialized (no model)
     - DecisionAgent: Category weights initialized (no model)
   - citizen_portal.py: CLIP model loaded via transformers.pipeline() and cached
   - admin_dashboard.py: No ML models loaded (uses only DB data and APIs)

5. Database Initialization:
   - DatabaseHandler connects to MongoDB Atlas with TLS
   - Connection verified via admin.command('ping')
   - is_connected flag set based on connection result
   - Collection reference established: civic_sentinel_db.grievances

6. Event Flow (Citizen Grievance Submission):
   a. Citizen authenticates via Firebase sidebar (login or signup)
   b. Citizen enters grievance text (or records voice, which is transcribed)
   c. Citizen optionally uploads image evidence
   d. If image and text present: CLIP performs zero-shot verification
   e. Citizen clicks "Get My Current Location" or uses manual coordinates
   f. Citizen clicks "Submit Individual Grievance"
   g. GrievanceOrchestrator.run_pipeline(text) executes 5-agent chain
   h. Data packet assembled: pipeline results + GPS + image + verification + metadata
   i. DatabaseHandler.submit_complaint(packet) inserts into MongoDB
   j. Success balloon animation and ticket ID displayed

7. Data Transformation Pipeline (Sequential):
   Step 1 - TRANSLATE: Raw text -> TranslationAgent -> English text
   Step 2 - CLASSIFY: English text -> ClassificationAgent -> Category + confidence
   Step 3 - SENTIMENT: English text -> SentimentAgent -> Compound score + polarity
   Step 4 - URGENCY: English text -> UrgencyAgent -> is_urgent boolean
   Step 5 - DECISION: (Category, sentiment, urgency) -> DecisionAgent -> CCI + priority

8. Response Generation (Admin Dashboard):
   a. All documents fetched from MongoDB on page load
   b. DataFrame constructed with hybrid GPS extraction
   c. Departments auto-assigned via keyword analysis
   d. KPI metrics calculated and rendered
   e. 3D heatmap generated with PyDeck ColumnLayer
   f. Analytics charts rendered with Altair
   g. Ticket resolution desk allows status updates + email notifications

9. Logging Execution:
   - Agent initialization logged to console: "Initializing AI Pipeline...", "BERT Model Loaded Successfully!", "Pipeline Ready."
   - DB connection logged: "DATABASE: Connected via MongoDB Atlas."
   - Email debug: detailed logging of recipient, ticket ID, remarks content
   - Timestamps logged on ticket updates

10. Error Handling Flow:
    - Critical errors (DB offline, model load failure): st.error() + st.stop()
    - Recoverable errors (translation failure, route unavailable): Fallback values + st.warning()
    - Validation errors (empty text, invalid email): st.warning() before submission
    - SMTP errors: Logged to console, ticket still updated in DB

11. Shutdown Behavior:
    - Streamlit servers terminate on SIGINT/SIGTERM
    - MongoDB connections cleaned up by pymongo client destructor
    - No explicit shutdown hooks or cleanup procedures implemented

12. Restart / Recovery Behavior:
    - Stateless application design: restart restores full functionality
    - All persistent data in MongoDB Atlas (survives restarts)
    - CLIP and BERT models re-loaded on restart (cached after first load)
    - No checkpointing or state recovery mechanism needed

13. Checkpoint Restoration:
    - Not applicable. No training occurs at runtime. All models are pre-trained.
    - Data persistence handled entirely by MongoDB Atlas (cloud-managed replication).
