SECTION 1 — PROJECT OVERVIEW
============================================================

Project Name: Civic Sentinel — Intelligent Citizen Grievance Prioritization System
Full Academic Title: Explainable Multi-Agent Decision Intelligence Framework for Context-Aware Citizen Grievance Prioritization (TARP)

High-Level Objective:
Civic Sentinel is an AI-powered civic grievance redressal system that receives citizen complaints (text, voice, image), processes them through a multi-agent intelligence pipeline, scores each complaint using a novel Civil Criticality Index (CCI), and presents prioritized results to government administrators via a real-time command center dashboard with geospatial routing capabilities.

Problem It Solves:
Traditional grievance redressal systems process complaints in FIFO order without regard for severity, urgency, or context. A minor noise complaint is treated identically to a life-threatening gas leak. Civic Sentinel solves this by applying AI-based triage: classifying, sentiment-analyzing, urgency-detecting, and priority-scoring every complaint before dispatching emergency resources via calculated optimal routes.

Target Users:
1. Citizens — Submit grievances through the Citizen Portal (text, voice, image).
2. Government Administrators — Monitor, triage, and resolve complaints through the Admin Command Center.
3. Emergency Response Units — Receive dispatches with optimal routing from nearest station.

Domain Category: AI/ML System + Civic Technology + Geospatial Intelligence Platform

System Type: Monolithic Multi-Agent Pipeline Architecture. The system is a single-deployment application composed of five specialized AI agents orchestrated by a central pipeline, with two Streamlit-based frontend interfaces and a cloud-hosted MongoDB backend.

Deployment Intention: Local development with cloud database (MongoDB Atlas) and cloud API integration (Mapbox Directions API, Google Translate API, Google Speech Recognition API). Production-ready for local or cloud deployment.

FILE 3.3: 3_agents/classification_agent/classifier.py
────────────────────────────────────────
File Location: Citizen-Grievance-Prioritization/3_agents/classification_agent/classifier.py
Purpose: AI agent that classifies citizen grievances into predefined municipal categories using a fine-tuned DistilBERT model.
Why This File Exists: Text classification is the foundational step in the pipeline — every downstream decision depends on knowing the grievance category.
Responsibilities: (1) Load fine-tuned DistilBERT model and tokenizer from local bert_brain/ directory. (2) Tokenize input text. (3) Perform forward-pass inference. (4) Convert logits to softmax probabilities. (5) Return predicted category label and confidence score.
Internal Components:
  - ClassificationAgent class:
    - __init__(): Loads DistilBertTokenizerFast and DistilBertForSequenceClassification from bert_brain/ directory. Sets model to eval() mode.
    - predict(text): Tokenizes input (max_length=128), performs torch.no_grad() inference, applies softmax, returns dict with 'category' and 'confidence'.
Input Sources: English text string from grievance_pipeline.py (post-translation).
Output Destinations: Returns dict {"category": str, "confidence": float} to grievance_pipeline.py.
Dependencies: torch, transformers (DistilBertTokenizerFast, DistilBertForSequenceClassification), os.
Runtime Behavior: Model loaded once during __init__. Each predict() call performs CPU inference in ~50-200ms. Model weights are ~268MB (model.safetensors).
Interaction with Other Files: Called by grievance_pipeline.py. Reads model files from bert_brain/ subdirectory.
Failure Handling: Returns {"category": "System Error", "confidence": 0.0} if model fails to load. Returns {"category": "Error", "confidence": 0.0} if prediction fails.

FILE 3.8: 3_agents/decision_agent/decision.py
────────────────────────────────────────
File Location: Citizen-Grievance-Prioritization/3_agents/decision_agent/decision.py
Purpose: Decision-making agent that computes the Civil Criticality Index (CCI) — the novel composite metric of the system — by synthesizing category, sentiment, and urgency signals.
Why This File Exists: CCI is the central innovation of the project. It fuses multi-agent outputs into a single actionable priority score (0-10).
Responsibilities: (1) Maintain category_weights dictionary mapping 11 grievance categories to base risk scores (3.0-9.0). (2) Apply urgency bonus (+1.5 or +2.0 depending on base severity). (3) Apply sentiment impact (negative sentiment adds up to +1.5). (4) Cap final score at 10.0. (5) Map score to priority label (High ≥7.5, Medium ≥4.5, Low <4.5).
Internal Components:
  - DecisionAgent class:
    - __init__(): Initializes category_weights dict for 11 categories.
    - compute_priority(category, sentiment_score, is_urgent): Returns dict with 'priority' (str) and 'cci' (float).
  CCI Formula: cci = min(base_score + urgency_bonus + sentiment_impact, 10.0)
    - base_score: Looked up from category_weights (default 4.0).
    - urgency_bonus: 2.0 if urgent AND base≥7.0; 1.5 if urgent AND base<7.0; 0.0 if not urgent.
    - sentiment_impact: abs(sentiment_score) * 1.5 if sentiment_score < 0; else 0.
Input Sources: Category (str), sentiment_score (float), is_urgent (bool) from grievance_pipeline.py.
Output Destinations: Returns dict {"priority": str, "cci": float} to pipeline.
Dependencies: None — pure Python logic.
Runtime Behavior: Deterministic mathematical computation. Sub-millisecond execution.
Interaction with Other Files: Called by grievance_pipeline.py. CCI score displayed across all UI surfaces.

SECTION 4 — SYSTEM ARCHITECTURE DESIGN
============================================================

Architectural Style:
Monolithic Multi-Agent Pipeline Architecture. The system is a single-deployment Python application using Streamlit as the web framework, with five specialized AI agents orchestrated by a central pipeline class (GrievanceOrchestrator). Two Streamlit applications (citizen_portal.py and admin_dashboard.py) share the same database and utility modules.

Component Separation Strategy:
The codebase follows a numbered-directory convention for logical separation:
  - 3_agents/: Individual AI agent modules, each in its own subdirectory.
  - 4_pipeline/: Orchestration layer that chains agents.
  - 7_mvp_application/: Standalone MVP without database dependency.
  - utils/: Shared utility modules (database, GPS, routing, PDF generation).
  - Root-level: Primary production applications (citizen_portal.py, admin_dashboard.py).

Service Boundaries:
Single process per Streamlit application. Two concurrent Streamlit servers run:
  1. citizen_portal.py — port 8501 (default) — citizen-facing.
  2. admin_dashboard.py — separate port — admin-facing.
Both share MongoDB Atlas as the persistence layer. No inter-process communication; synchronization happens through shared database state.

Data Flow Design:
  Citizen Input (Text/Voice/Image)
    → [citizen_portal.py] Voice Transcription (Google Speech API)
    → [citizen_portal.py] Image Verification (CLIP zero-shot classification)
    → [grievance_pipeline.py] Translation (Google Translate via deep_translator)
    → [grievance_pipeline.py] Classification (Fine-tuned DistilBERT)
    → [grievance_pipeline.py] Sentiment Analysis (NLTK VADER)
    → [grievance_pipeline.py] Urgency Detection (Regex keyword + negation)
    → [grievance_pipeline.py] Decision (CCI calculation)
    → [citizen_portal.py] GPS Attachment + Database Submission (MongoDB Atlas)
    → [admin_dashboard.py] Fetch All → Auto-Department Assignment → KPI Metrics
    → [admin_dashboard.py] 3D Geospatial Heatmap (PyDeck)
    → [admin_dashboard.py] Navigator Route Calculation (Mapbox API)
    → [admin_dashboard.py] Ticket Resolution + Email Notification (SMTP/Gmail)

State Management Strategy:
  - Session-Level: Streamlit st.session_state stores cached model instances, batch data, and UI state (GPS coordinates, form values).
  - Persistent: MongoDB Atlas stores all submitted complaints with server timestamps.
  - Cached Resources: @st.cache_resource decorators prevent re-loading ML models and database connections across Streamlit reruns.

Model Lifecycle:
  1. DistilBERT (Classification): Loaded from local bert_brain/ directory at first request. Cached via @st.cache_resource. Read-only inference.
  2. CLIP (Image Verification): Loaded from Hugging Face Hub at first request. Cached. ~600MB download on first run.
  3. VADER (Sentiment): Lexicon file downloaded via nltk.download() on first use. Cached in NLTK data directory.

Configuration Management:
  - API secrets stored in .streamlit/secrets.toml (Mapbox token).
  - Email credentials hardcoded in admin_dashboard.py (SENDER_EMAIL, APP_PASSWORD).
  - MongoDB URI hardcoded in utils/db_handler.py.
  - Emergency station coordinates hardcoded in utils/route_engine.py and admin_dashboard.py.
  - No environment variable loading in production use.

Logging Strategy:
  - Console-based logging via print() statements throughout all modules.
  - Formatted with emoji prefixes for visual scanning (✅, ⚠️, ❌, 🔄).
  - No structured logging framework (no logging module, no log files).
  - Admin dashboard displays debug expanders for GPS telemetry and route data.

Error Handling Strategy:
  - Try/except blocks at every external boundary (API calls, database operations, model inference).
  - Graceful degradation: Database offline → system continues in read-only/offline mode.
  - UI-level error display via st.error(), st.warning(), st.toast().
  - Fallback values for all agent outputs (e.g., "System Error" category, 0.0 confidence).
  - Zero-crash policy enforced in admin_dashboard.py with mandatory data validation.

Scalability Considerations:
  - Horizontal: Not currently designed for horizontal scaling. Single-instance Streamlit servers.
  - Vertical: BERT inference is CPU-bound; GPU would improve classification throughput.
  - Database: MongoDB Atlas provides built-in horizontal scaling and replication.
  - Bottleneck: Sequential pipeline execution — all 5 agents run serially per complaint.

Security Considerations:
  - MongoDB Atlas connection uses TLS/SSL via certifi.
  - Mapbox token stored in secrets.toml (not committed to Git via .gitignore).
  - Email credentials (Gmail App Password) are hardcoded — security risk in production.
  - No authentication/authorization on Streamlit interfaces.
  - No input sanitization beyond CLIP verification and email format check.
  - Image data stored as base64 string in MongoDB — no file upload vulnerability.

SECTION 5 — COMPLETE END-TO-END WORKFLOW
============================================================

STEP 1: System Initialization
  Both Streamlit applications (citizen_portal.py and admin_dashboard.py) are started as separate processes via:
    $ streamlit run citizen_portal.py
    $ streamlit run admin_dashboard.py

STEP 2: Configuration Loading
  - Streamlit reads .streamlit/secrets.toml automatically at startup.
  - Mapbox token becomes available via st.secrets["general"]["mapbox_token"].

STEP 3: Model Loading (First Request Only)
  citizen_portal.py:
    3a. @st.cache_resource loads CLIP pipeline (openai/clip-vit-base-patch32) — ~600MB, 10-30s first load.
    3b. @st.cache_resource instantiates GrievanceOrchestrator:
      → TranslationAgent: GoogleTranslator initialized (no model download).
      → ClassificationAgent: DistilBERT loaded from bert_brain/ (~268MB), set to eval().
      → SentimentAgent: VADER lexicon downloaded if missing.
      → UrgencyAgent: Keyword lists initialized (no external resources).
      → DecisionAgent: Category weights dict initialized.
    3c. @st.cache_resource connects DatabaseHandler to MongoDB Atlas.
  admin_dashboard.py:
    3d. @st.cache_resource connects DatabaseHandler to MongoDB Atlas.

STEP 4: Database Initialization
  - DatabaseHandler.__init__() creates pymongo.MongoClient with 5-second timeout and TLS.
  - Pings MongoDB Atlas via client.admin.command('ping').
  - Accesses database "civic_sentinel_db" and collection "grievances".
  - Sets is_connected = True on success, False on failure.

STEP 5: Citizen Submits Grievance (citizen_portal.py)
  5a. Citizen enters email and grievance text.
  5b. Optional: Uploads image → CLIP verifies relevance against text (>30% match = Verified, else Flagged).
  5c. Optional: Records voice → Google Speech Recognition transcribes to text.
  5d. GPS: Browser geolocation fetched via streamlit-js-eval, or manual coordinates.
  5e. Citizen clicks "Submit Individual Grievance".

STEP 6: Pipeline Execution (grievance_pipeline.py)
  6a. TranslationAgent.translate(text): Auto-detects language, translates to English.
  6b. ClassificationAgent.predict(english_text): DistilBERT classifies into 1 of 11 categories.
  6c. SentimentAgent.analyze(english_text): VADER computes compound sentiment score.
  6d. UrgencyAgent.analyze_urgency(english_text): Regex checks negation patterns, then urgent keywords.
  6e. DecisionAgent.compute_priority(category, sentiment, urgency): Computes CCI score and priority label.
  6f. All results merged into flat dictionary with 14+ fields.

STEP 7: Data Persistence
  7a. Pipeline results merged with metadata (email, GPS, image data, verification score, status="New").
  7b. DatabaseHandler.submit_complaint() adds server_timestamp and status="Pending".
  7c. Document inserted into MongoDB Atlas "grievances" collection.
  7d. Inserted ObjectId returned as ticket ID to citizen.

STEP 8: Admin Dashboard Data Pipeline (admin_dashboard.py)
  8a. DatabaseHandler.fetch_all_complaints() retrieves all documents.
  8b. Raw data converted to Pandas DataFrame.
  8c. ObjectIds converted to strings.
  8d. Numeric columns (cci, sentiment_score) coerced to float.
  8e. Text extracted from multiple possible field names.
  8f. GPS coordinates extracted from flat or nested structures.
  8g. Departments auto-assigned via keyword analysis of category + text.
  8h. Sidebar filters applied.

STEP 9: Visualization and Analytics
  9a. KPI metrics rendered (total tickets, critical ops, avg response, mean risk).
  9b. 3D geospatial heatmap rendered via PyDeck ColumnLayer (elevation = CCI * 100).
  9c. Arc vectors rendered for single-department dispatch mode.
  9d. Altair charts: priority donut, sentiment scatter, risk-by-unit bar.

STEP 10: Ticket Resolution
  10a. Admin selects ticket from dropdown.
  10b. Incident details displayed (description, language, image evidence indicator).
  10c. Navigator calculates route: find_nearest_station() → get_route_geometry() → PyDeck PathLayer.
  10d. Admin updates status (Pending/In Progress/Resolved/Rejected) and adds remarks.
  10e. MongoDB document updated with new status, admin_comment, and timestamp.

STEP 11: Email Notification (Echo System)
  11a. If citizen email exists in ticket data, send_notification_email() triggered.
  11b. Text cleaned of non-ASCII characters.
  11c. MIME multipart email constructed (plain text + HTML template).
  11d. Sent via Gmail SMTP (smtp.gmail.com:587, STARTTLS).
  11e. Success/failure feedback shown to admin.

STEP 12: Shutdown Behavior
  Streamlit processes terminate via Ctrl+C or process kill. No graceful shutdown hooks. MongoDB connections closed by pymongo destructor.

STEP 13: Restart / Recovery
  Full system restart by re-running streamlit commands. All state recovered from MongoDB Atlas. Cached models reloaded on first request. No checkpoint files.

SECTION 6 — DEPLOYMENT & EXECUTION GUIDE
============================================================

Environment Setup:
  1. Python 3.9+ required (3.10 or 3.11 recommended).
  2. Clone repository or copy project directory.
  3. Create virtual environment:
       $ python -m venv .venv
       $ source .venv/bin/activate

Dependency Installation:
  $ pip install -r requirements.txt
  Additional dependencies not in requirements.txt:
  $ pip install deep-translator nltk fpdf certifi

NLTK Data Download (automatic on first run, or manual):
  $ python -c "import nltk; nltk.download('vader_lexicon')"

BERT Model:
  The bert_brain/ directory must contain model.safetensors, config.json, tokenizer.json, tokenizer_config.json. These files are excluded from Git via .gitignore and must be obtained separately (from training output or shared storage).

Environment Variables / Secrets:
  1. .streamlit/secrets.toml must contain:
     [general]
     mapbox_token = "YOUR_MAPBOX_PUBLIC_TOKEN"
  2. In admin_dashboard.py, update:
     SENDER_EMAIL = "your_gmail@gmail.com"
     APP_PASSWORD = "your_gmail_app_password"
  3. In utils/db_handler.py, update MONGO_URI with your MongoDB Atlas connection string.

Running Locally:
  Terminal 1 (Citizen Portal):
    $ streamlit run citizen_portal.py
  Terminal 2 (Admin Dashboard):
    $ streamlit run admin_dashboard.py
  Terminal 3 (Standalone MVP — no database required):
    $ streamlit run 7_mvp_application/app.py

Docker Deployment:
  Not currently containerized. To dockerize:
  1. Create Dockerfile based on python:3.11-slim.
  2. COPY project files, install requirements.
  3. EXPOSE ports 8501 and 8502.
  4. CMD: streamlit run citizen_portal.py & streamlit run admin_dashboard.py

Testing:
  $ python test_clip.py         # Test CLIP model inference speed
  $ python reset_db_test.py     # Test MongoDB connectivity and reset
  $ python 4_pipeline/grievance_pipeline_OLD.py  # Test legacy pipeline standalone

Monitoring:
  - Console output with emoji-prefixed log messages.
  - Admin dashboard Debug expanders for GPS telemetry and route data.
  - MongoDB Atlas dashboard for database monitoring.
  - Streamlit built-in server logs.

Production Deployment:
  1. Replace hardcoded credentials with environment variables.
  2. Add authentication (Streamlit authentication or reverse proxy).
  3. Enable HTTPS via reverse proxy (Nginx/Caddy).
  4. Deploy on cloud VM (AWS EC2, GCP Compute Engine) or Platform-as-a-Service.
  5. Use process manager (systemd, supervisord) for Streamlit servers.
  6. Configure MongoDB Atlas IP whitelist for production IP.

System Identity:
  - Name: Civic Sentinel

  1. TranslationAgent (translator.py) — Google Translate auto-detect → English
  2. ClassificationAgent (classifier.py) — Fine-tuned DistilBERT → 11 categories
  3. SentimentAgent (sentiment.py) — NLTK VADER → compound score (-1 to +1)
  4. UrgencyAgent (urgency_detect.py) — Regex keyword + negation → boolean
  5. DecisionAgent (decision.py) — CCI = base_weight + urgency_bonus + sentiment_impact → priority label
  6. GrievanceOrchestrator (grievance_pipeline.py) — Sequential orchestrator: 1→2→3→4→5
  7. DatabaseHandler (db_handler.py) — MongoDB Atlas CRUD operations
  8. LocationManager (location_manager.py) — Browser GPS via streamlit-js-eval
  9. RouteEngine (route_engine.py) — Mapbox Directions API + haversine nearest-station
  10. ReportGenerator (report_generator.py) — FPDF-based PDF situation reports
  11. CitizenPortal (citizen_portal.py) — Multimodal input + CLIP verification + batch upload + tracking
  12. AdminDashboard (admin_dashboard.py) — 3D heatmap + Navigator routing + Echo emails + resolution desk

Data Flow (Short Form):
  User Input → Translation → BERT Classification → VADER Sentiment → Urgency Keywords → CCI Decision → MongoDB → Admin Fetch → Auto-Dispatch → PyDeck Map → Navigator Route → Status Update → Email Notification

Critical Dependencies:
  - torch + transformers: DistilBERT inference (classification_agent)
  - deep_translator: Google Translate API (translation_agent)
  - nltk: VADER sentiment lexicon (sentiment_agent)
  - pymongo + certifi: MongoDB Atlas connectivity (db_handler)
  - streamlit + streamlit-js-eval: Web UI + browser geolocation
  - pydeck: 3D map visualization (admin_dashboard)
  - requests: Mapbox Directions API (route_engine)
  - fpdf: PDF report generation (report_generator)
  - Pillow: Image processing for CLIP verification
  - speech_recognition: Google Speech API for voice input
  - smtplib: Gmail SMTP for citizen notifications

Core Execution Sequence:
  1. Start Streamlit servers (citizen_portal.py, admin_dashboard.py)
  2. Load secrets.toml (Mapbox token)
  3. Cache-load models (CLIP, DistilBERT, VADER)
  4. Connect MongoDB Atlas (DatabaseHandler)
  5. Citizen submits grievance (text/voice/image + GPS)
  6. Pipeline: Translate → Classify → Sentiment → Urgency → CCI Decision

  11. Select ticket → Navigator calculates route from nearest station
